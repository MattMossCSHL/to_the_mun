# handoff 2026-04-03

## where things stand

The project made real progress upstream and in the KSP harness today.

The GA is no longer being dominated by two major analytic scoring bugs:

- launch TWR is now based on the actual liftoff engine rather than summed whole-rocket thrust
- stage delta-v / burn-time now require a valid full propellant mixture rather than crediting partial matches

After those fixes, the new rank-1 rocket from:

- `data/runs/run_2026-04-03-121004/gen_020.json`

looked analytically plausible and launched through the KSP harness.

## important code state

This closed:

### serializer

`src/craft.py` now resolves:

- `Rockomax64_BW` -> `Rockomax64.BW`

via improved craft-name alias handling for uppercase suffixes.

That removed the fallback-block problem for the top test rocket's large tank.

### runner / mission code

`scripts/run_saved_rocket.py` was changed substantially:

- passive rails warp added for:
  - coast to Kerbin apoapsis
  - Mun transfer-window wait
  - coast to Mun SOI
  - coast to Mun periapsis
- phase handoff and upper-stage ignition were separated:
  - handoff now stages/separates and stabilizes only
  - ignition remains in active burn phases
- handoff rule was tightened repeatedly and ended the session in the most conservative state:
  - `apoapsis >= target_apoapsis`
  - `periapsis >= 0`
  - remaining monitored lower-stage propellant `<= handoff_fuel_threshold`
- `phase_angle(...)` was corrected to use the `X/Z` equatorial plane of KSP's non-rotating frame instead of `X/Y`

The runner compiles after these changes:

```bash
python3 -m py_compile scripts/run_saved_rocket.py
```

## addendum after later serializer work

This handoff needs one important correction.

The later `Size2LFB` candidate should **not** be treated as invalid simply because it used that part.

What was actually proven in the follow-up work:

- `Size2LFB` is a valid stock KSP part and can be kept in play
- the real serializer bug was that `src/craft.py` still used a fake generic fallback path for parts without a `.craft` template
- a separate template-leak bug copied `VESSELNAMING` from stock craft and caused generated vessels to appear as `Ariane 5`

### serializer state at true session end

`src/craft.py` now does the following:

- use craft templates when available
- otherwise resolve the stock `part.cfg` from `_source_file`
- parse the stock `PART` block with `parse_cfg(...)`
- copy real `RESOURCE` and `MODULE` blocks into the generated craft
- strip leaked `VESSELNAMING` blocks from copied templates

It also now handles the earlier linear-stack node assumptions more defensively:

- child-side attach node is chosen from the part's actual available nodes instead of assuming `top`
- parent-side attach node is chosen from available nodes instead of trusting an absent requested node

This was enough to serialize the current rank-1 rocket from:

- `data/runs/run_2026-04-03-145251/gen_020.json`

with:

- template-backed parts:
  - `probeStackSmall`
  - `Rockomax8BW`
  - `liquidEngine3_v2`
  - `Decoupler_0`
  - `Size3LargeTank`
- cfg-backed part:
  - `Size2LFB`
