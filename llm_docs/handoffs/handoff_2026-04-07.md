# handoff 2026-04-07

## where things stand

This session materially improved the KSP runner and narrowed the mission failure to a specific upper-stage runtime problem.

The mission is no longer primarily blocked by:

- serializer dishonesty
- missing upper-stage hardware
- accidental destructive over-staging during orbital insertion
- missed Mun transfer windows
- stage-accounting blindness from exclusive-only decouple-stage resource queries

The mission now gets much farther and fails more honestly.

## most important outcome

The surviving upper stage is real, fueled, and ignited, but still produces **no actual thrust** during the Mun transfer burn.

Late-session evidence from the runner showed:

- surviving vessel after handoff:
  - `Rockomax8BW`
  - `liquidEngine3.v2`
  - `probeStackSmall`
- vessel-wide propellant on that surviving craft:
  - `LiquidFuel: 360`
  - `Oxidizer: 440`
- runtime engine state for the Terrier:
  - `active = True`
  - `has_fuel = True`
  - `available_thrust = 60000`
  - `actual_thrust = 0.0`
  - engine `throttle = 0.0`

The final mission failure of the day was:

- `mission_phase = mun_transfer_burn_no_actual_thrust`

So the active blocker is now:

- **runtime engine behavior / throttle / activation state on the surviving Terrier upper stage**

not transfer timing, not missing fuel, and not missing hardware.

## major improvements made this session

### ascent / handoff reset

`scripts/run_saved_rocket.py` was rewritten toward a cleaner explicit phase model:

- `liftoff_ascent`
- `core_stage_finish`
- `coast_to_apoapsis`
- `orbital_insertion`

The old accumulated handoff logic was replaced with a simpler, more inspectable model plus clearer event logging.

### insertion / staging fixes

- fixed destructive staging behavior in later burn phases by refusing to auto-stage into a dead final stage
- added explicit one-time upper-stage ignition at orbital insertion start
- added explicit ignition at Mun transfer start when needed

### transfer-window fixes

- reduced warp aggressiveness near the Mun transfer window
- added sign-cross capture so the window can be treated as reached even if warp steps across the exact tolerance boundary

### burn-guidance fix

Vacuum orbital maneuvers now use:

- `vessel.orbital_reference_frame`

instead of:

- `vessel.surface_velocity_reference_frame`

This changed:

- orbital insertion guidance
- Mun transfer-burn guidance
- Mun capture guidance

### resource/accounting fixes

The old stage-resource accounting was too narrow.

Changes made:

- `stage_resource_amount(...)` now defaults to cumulative stage access
- `stage_resource_breakdown(...)` also defaults to cumulative access
- diagnostics now show both:
  - cumulative
  - exclusive
- vessel-wide propellant breakdowns were added because they proved more trustworthy than stage-bucket views for this craft

### runtime diagnostics added

The runner now records:

- vessel composition snapshots
- removed parts at stage events
- propulsion snapshots with:
  - vessel-wide resource totals
  - stage breakdowns
  - available thrust
  - actual thrust
  - mass
- engine runtime snapshots when available through kRPC, including:
  - active
  - has_fuel
  - thrust
  - available_thrust
  - throttle
  - stage / decouple stage

Verbose logging was also reduced to more change-driven, timestamped output.

## static inspection findings

Static inspection did **not** reveal an obvious impossible-craft geometry issue for the surviving upper stage.

What was checked:

- the surviving upper-stage stack geometry
- the generated Terrier PART block
- comparison against a stock craft Terrier PART block

Current conclusion:

- the generated Terrier PART block is structurally close to a stock one
- nothing obvious in the static craft text screamed "this engine can never work"

That does **not** prove the craft text is perfect, but it means the next session should focus first on runtime engine behavior rather than restarting serializer work from zero.
