## design decisions locked in

- **population** = list of `(rocket_dict, meta)` tuples. `meta` always has `score`; `detailed=True` adds `valid`, `filtered`, `n_stages`, `n_parts`, `stage_dv`, `generation`
- **fitness** = delta-v or 0. cliff at 0, gradient above.
- **selection** = tournament (with replacement, `pct_survivors=0.5`, `tournament_size=3`)
- **crossover** = stage-level. random cut on A, graft B's booster stages. returns A unchanged if B is single-stage.
- **elitism** = top `n_elites=5` carry forward unchanged to prevent regression
- **mutation rate** = 0.3 default. each child mutated with this probability post-crossover.
- **per-generation export** = `gen_NNN.json` in `save_dir` if provided
- **progressive complexity** (increase max_stages on threshold) ‚Äî deferred, see design_decisions.md
- **radial attachment** ‚Äî deferred, later milestone
- **`Rocket.from_dict()`** ‚Äî deferred, note in `src/rocket.py`

- probe cores appear in `tanks` list ‚Äî pod exclusion by id essential in `mutate_swap_part`

## what's next

the GA is complete and working. the likely next step is **building the KSP harness** ‚Äî the interface that takes top candidate rockets from the GA, simulates them in KSP, and collects ground-truth flight data.

this is the bridge between the analytic pipeline (fast, approximate) and the simulation oracle (slow, accurate). key questions to figure out:
- how does a rocket dict get translated into an actual KSP craft file?
- how do we automate launching and reading telemetry from KSP?
- what data do we collect per flight (apoapsis, delta-v achieved, staging events, etc.)?
- how does the collected dataset feed back into training a surrogate model?

secondary next steps:
- **surrogate model** ‚Äî once KSP data exists, train a model to predict simulation outcomes from rocket dicts
- **progressive complexity** ‚Äî add `max_stages` growth on score threshold (see design_decisions.md)
- **demo notebook** ‚Äî user is creating `notebooks/demo.ipynb` with the three-cell setup/run/plot block
