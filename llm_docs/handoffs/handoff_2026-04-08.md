# handoff 2026-04-08

## where things stand

This session changed the project-level diagnosis in an important way.

The runner-side Terrier problem was re-tested, but the more important outcome was upstream:

- the current GA/design-space setup can still produce analytically strong but mission-incoherent rockets
- one clear cause was unmanned probe roots with no power-support parts in the design space
- another clear cause is the absence of a geometry/stability filter for obviously unstable inline stacks

## most important outcomes

### 1. probe roots were removed from the current pod pool robustly

The temporary heuristic idea was rejected in favor of a cleaner data-model change.

What changed:

- `src/scraper.py` now extracts `crew_capacity`
- `data/parts_library.json` was regenerated
- `src/config.py::load_part_lists(...)` now includes only command parts with `crew_capacity > 0` in `pods`

This means:

- probe cores such as `probeStackSmall` are no longer eligible as `pod_0`
- crewed command parts remain available

This was the right near-term decision because the current GA design space does **not** include batteries, solar, or RTGs, so supporting unmanned Mun-capable rockets would be dishonest.

### 2. the parts library was extended with geometry/aero metadata

Before building a geometry filter, raw Squad cfg inspection was done to verify what extra information exists in source data.

Useful fields were found and are now scraped into the main `data/parts_library.json`:

- `bulkhead_profiles`
- `com_offset`
- `cop_offset`
- `col_offset`
- `drag`
- `breaking_force`
- `breaking_torque`
- `vessel_type`

Representative checks confirmed the regenerated library is good:

- `Size3To2Adapter_v2`
  - `bulkhead_profiles = ['size2', 'size3']`
  - drag metadata present
  - break-force/torque metadata present
- `Mark1Cockpit`
  - `crew_capacity = 1`
  - `com_offset`, `cop_offset`, `col_offset` present
  - `vessel_type = 'Plane'`
- `probeStackSmall`
  - `crew_capacity = 0`
  - `vessel_type = 'Probe'`

## what this means technically

The project now has enough information in the main parts library to build a serious first-pass inline geometry filter without pretending to have full KSP aerodynamics.

Available now:

- dry mass
- stored resources
- top/bottom node positions
- top/bottom node sizes
- crew capacity
- bulkhead profiles
- CoM / CoP / CoL offsets when declared
- simple drag metadata

That is enough for a first-pass filter based on:

- stack height
- diameter profile
- diameter-transition penalties
- approximate axial center of mass
- top-heaviness
- slenderness ratio

It is **not** enough for true aerodynamic stability, but it is enough to reject obviously unstable inline stacks far more honestly than blind GA tuning.

## what to do first next session

Build the geometry filter.

Recommended order:

1. define a helper that reconstructs an inline stack’s axial geometry from the rocket dict plus `parts_by_name`
2. compute first-pass metrics:
   - total height
   - per-part diameter class / radius profile
   - number and severity of diameter transitions
   - approximate axial CoM
   - slenderness ratio
   - top-heaviness metric
3. decide which of those should be:
   - hard rejects
   - soft penalties
4. apply the filter before doing more GA parameter tuning

## things that should *not* be the first move next session

Do **not** begin by:

- tuning mutation rate again
- tuning `n_elites` again
- retuning the runner for the unstable five-stage solid/adaptor rockets
- banning `Size3To2Adapter_v2` just because it appeared frequently

Those are downstream reactions. The cleaner next move is to filter the failure mode, not the part.
