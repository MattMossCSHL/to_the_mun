## most important outcomes

### 1. geometry filter was built and extracted

Work was done first in `notebooks/structural_validator.ipynb`, then extracted to `src/structure.py` with docstrings only.

Added:

  - `build_part_lookup(...)`
  - `build_children_lookup(...)`
  - `get_root_part_id(...)`

- geometry aggregation:
  - `compute_geometry_metrics(...)`
- geometry gate:
  - `check_geometry_filter(...)`

`validate_rocket(...)` in `src/structure.py` now runs the geometry filter before the older structural checks.

### 2. diameter proxy was corrected

Raw node sizes were too misleading for geometry checks.

The diameter proxy now prefers scraped `bulkhead_profiles` with approximate KSP stack diameters:

- `size0 = 0.625`
- `size1 = 1.25`
- `size1p5 = 1.875`
- `size2 = 2.5`
- `size3 = 3.75`

Node sizes are used only as a fallback.

### 3. stack-safety moved into the scraped parts library

The old stack-safety heuristic in `load_part_lists(...)` only excluded lowercase `mk2` / `mk3`, which missed names like `Mark2Cockpit`.

Fix:

- `src/scraper.py` now writes:
  - `stack_safe`
- current scrape-time heuristic excludes names containing:
  - `mk2`
  - `mk3`
  - `mark2`
  - `mark3`
- `src/config.py::load_part_lists(...)` now trusts `part['stack_safe']`
- `data/parts_library.json` was regenerated

### 4. category loopholes in the GA part pools were cleaned up

The old pool classification rules were too broad.

Problems found:

- `tanks` admitted wings, probes, radial tanks, and fuel adapters
- `engines` admitted jets, ions, radial engines, OMS thrusters, and the LES
- `pods` admitted `Mark1Cockpit`

`src/config.py::load_part_lists(...)` was tightened so the simple inline rocket pools are now:

- `pods`
  - crewed only
  - stack-safe only
  - non-`Plane`
  - must have both `top` and `bottom` nodes
- `tanks`
  - no engine
  - not command parts
  - category in `FuelTank` / `Propulsion`
  - must carry supported tank propellants
  - must have both `top` and `bottom` nodes
  - must have matching top/bottom node sizes so adapters do not count as tanks
- `engines`
  - must have engine data
  - category in `Engine` / `none`
  - propellants limited to `{LiquidFuel, Oxidizer, SolidFuel}`
  - must have both `top` and `bottom` nodes

Verified exclusions included:

- `Mark2Cockpit`
- `Mark1Cockpit`
- `Size3To2Adapter_v2`
- `adapterSize2-Size1Slant`
- `MpoProbe`
- `turboJet`
- `ionEngine`
- `omsEngine`
- `LaunchEscapeSystem`
- `smallRadialEngine`

### 5. GA outputs became cleaner

After reloading the updated part pools in the notebook, the GA stopped converging on obvious category exploits.

Cleaner later runs looked like:

- normal rocket pods such as `mk1-3pod` / `cupola`
- real tanks such as `fuelTankSmall` / `Size3SmallTank`
- real inline rocket engines such as `engineLargeSkipper_v2`, `nuclearEngine`, `toroidalAerospike`

This does **not** mean the search space is perfect, but it is much more honest than before.

### 6. plot overlay was improved

`src/plots.py::plot_run(...)` now draws a blue dotted best-score-per-generation line.

That line is clipped to the same `y_cap` as the scatter so it remains visible even when best scores exceed the 99th-percentile display cap.

### 7. runner staging logic was debugged and partially fixed

Two distinct runner issues were found and patched in `scripts/run_saved_rocket.py`.

#### fix 1: stage-number mapping for resource tracking

The `.craft` staging itself was consistent, but the runner was inferring the wrong KSP decouple-stage buckets for resource monitoring.

Old mapping:

- `0 -> 0`
- `n > 0 -> 2 * n`

Corrected mapping:

- `0 -> 0`
- `n > 0 -> 2 * n - 1`

This aligned designed stage propellants with the live KSP resource buckets seen through `resources_in_decouple_stage(...)`.

#### fix 2: stage-empty detection was using the all-stage propellant union

This was the more important live-run bug.

The runner was monitoring each stage against the union of all propellants across all stages. That let a stage look "non-empty" because of fuel belonging to some other attached stage.

Fix:

- added stage-specific propellant resolution
- `stage_resource_amount(...)` and `stage_resource_breakdown(...)` now support stage-specific propellant maps
- `current_fueled_stage(...)`, `maybe_stage(...)`, `handoff_to_upper_stage(...)`, and `has_lower_fueled_stage(...)` now use per-stage propellant sets via `stage_state`

## latest live run and current blocker

The most recent live test was:

- `gen_050_rank_001`
- command:
  - `python3 scripts/run_saved_rocket.py --generation-file .../gen_050.json --rank 1 --mission mun_orbit ...`

The test craft was:

- `cupola`
- `fuelTankSmall`
- `nuclearEngine`
- `Decoupler_1`
- `Rockomax16_BW`
- `microEngine_v2`
- `Decoupler_3`
- `externalTankToroid`
- `MassiveBooster`

The new log showed:

- `designed stage propellants: {0: ('LiquidFuel',), 1: ('LiquidFuel', 'Oxidizer'), 3: ('SolidFuel',)}`
- initial ascent:
  - `stage=4`
  - `fuel_stage=3`
  - booster fuel tracked correctly
- at booster depletion:
  - `fuel_empty_stage: begin current_stage=3`
- after the event:
  - `stage_after=2`
  - `fuel_stage=1`

That is materially better than the earlier behavior.

However, the run still did **not** become healthy.

Key ambiguity:

- the post-stage `remaining=` snapshot only showed the upper stack:
  - `Decoupler.1`
  - `Rockomax16.BW`
  - `cupola`
  - `fuelTankSmall`
  - `microEngine.v2`
  - `nuclearEngine`
- that suggests the lower booster stack may actually have detached
- but the same log still said:
  - `removed=none`

So the current state is:

- the old fuel-bucket bug is probably no longer the main blocker
- the newest blocker is now likely one of:
  - a logging/event-selection bug that hides the real removed-part delta
  - bad post-separation engine / thrust / throttle state
  - a truly non-viable upper stage (`microEngine_v2` + `Rockomax16_BW`) that stalls immediately after booster separation

One important clue:

- after the stage event, apoapsis dropped from about `15283m` to `14430m`
- so even if detachment occurred, the post-separation ascent performance was not good
