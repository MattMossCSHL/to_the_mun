# genetic algorithm primer

written for someone comfortable with computational methods and optimization but new to evolutionary algorithms.

---

## the core idea

a genetic algorithm (GA) is a search algorithm inspired by biological evolution. you maintain a **population** of candidate solutions, repeatedly select the best ones, combine and mutate them to produce new candidates, and iterate until you find something good enough.

for this project: candidates are rocket designs. "good enough" means clearing 70 km apoapsis (Goal 1), or eventually reaching orbit, the Mun, etc.

the analogy to biology is intentional and fairly deep:
- **genotype** — the rocket dict (the representation)
- **phenotype** — the rocket's actual flight performance (what KSP produces)
- **fitness** — how well the rocket achieves the mission goal
- **selection** — keep the better-performing designs
- **crossover** — combine two designs to produce offspring
- **mutation** — randomly modify a design

GAs are not gradient-based. they don't compute derivatives. they explore the fitness landscape by sampling it — generating candidates, evaluating them, and biasing future generation toward what worked. this makes them well-suited for problems where:
- the search space is discrete and combinatorial (which parts, in what arrangement)
- the fitness function is expensive (KSP simulation)
- the fitness landscape is rugged (small changes can have large effects)

all three are true here.

---

## the algorithm, step by step

```
1. initialize — generate a population of N random valid rocket designs
2. evaluate  — score each design (run through validator → filters → KSP if it passes)
3. select    — keep the top-performing designs (survivors)
4. reproduce — generate new designs from survivors via crossover and mutation
5. repeat    — go to step 2 until stopping condition met
```

this loop is called a **generation**. you run many generations. over time, the population drifts toward designs that score well.

---

## initialization

generate N random rockets that pass the structural validator and analytic filters. this is already most of the work — the `Rocket` class and `validate_rocket` handle structural validity, and the filters handle physics plausibility.

a random valid rocket might be generated by:
1. pick a random command pod
2. pick a random number of fuel tank + engine pairs and stack them
3. assign staging
4. run through validator and filters — if it fails, try again

this is rejection sampling. most random rockets will fail (wrong propellants, TWR < 1, etc.), but the filters are cheap so this is fine.

population size N is a hyperparameter. start small (50–100) while developing, scale up later.

---

## fitness function

the fitness function maps a rocket design to a scalar score. this is the most important design decision in the GA.

for Goal 1 (exit atmosphere, apoapsis >= 70 km), options include:
- **binary**: 1 if apoapsis >= 70 km, 0 otherwise — simple but gives no gradient signal
- **continuous**: apoapsis in km — a rocket that reaches 60 km scores better than one that reaches 10 km, even if neither passes
- **penalized**: apoapsis - penalties for mass, part count, cost — encourages efficient designs

continuous fitness is almost always better than binary for GA convergence. binary gives the GA no signal to follow when no design in the population passes yet.

for this project: use apoapsis (km) as the primary fitness signal. the GA has something to climb even before any design clears 70 km.

---

## selection

after evaluating the population, you select which designs survive to reproduce. common strategies:

**truncation selection** — keep the top K% of the population by fitness. simple and aggressive. fast convergence but low diversity.

**tournament selection** — randomly pick M candidates, keep the best one. repeat to fill the survivor pool. preserves more diversity than truncation.

**fitness-proportionate (roulette wheel)** — probability of surviving is proportional to fitness. designs with high fitness get more reproductive chances. can be unstable if fitness values are very skewed.

for this project: tournament selection is a reasonable default. it's robust and easy to implement.

---

## crossover

crossover combines two parent designs to produce offspring. the idea is that if two different designs each have something good about them, their offspring might inherit both good things.

for rocket designs, crossover is non-trivial because rockets have structure (a tree). naive crossover (split the parts list in half and swap) can produce invalid designs. sensible options:

**stage crossover** — take stages from parent A and stages from parent B. e.g. upper stage from A, booster stage from B. preserves valid stage structure.

**subtree crossover** — pick a random subtree (e.g. everything below a decoupler) from each parent and swap them. preserves local validity within subtrees.

**parameter crossover** — keep the same structure but blend parameters (e.g. average the number of tanks between two designs). only works if designs have compatible topologies.

for this project: stage crossover is the most natural given our part tree and staging dict structure. implement it after mutation is working.

---

## mutation

mutation randomly modifies a design. it's the primary source of novelty — crossover recombines what already exists, mutation introduces new material.

for rocket designs, sensible mutations include:
- **swap engine** — replace an engine with a different one of similar size
- **add tank** — insert a fuel tank above an engine
- **remove tank** — remove a tank from a stack
- **change staging** — reassign a stage number
- **add stage** — add a new booster stage below an existing one
- **remove stage** — drop the lowest stage

each mutation must be followed by a validity check. if the mutation produces an invalid rocket, discard it and try again (or try a different mutation).

mutation rate is a hyperparameter — too high and you're doing random search; too low and the population stagnates. start around 10–20% per design per generation.

---

## the fitness landscape for rockets

rocket design has a particular structure worth understanding before building the GA.

**staging is a discrete cliff**: adding a stage doesn't gradually improve performance — it either works (big delta-v gain) or it breaks the design (staging failure). this means the fitness landscape has discrete jumps, which GAs handle better than gradient methods.

**mass ratio dominates**: the rocket equation says delta-v depends on ln(m_wet/m_dry). this means there's strong selection pressure toward high mass ratios — lots of fuel, light structure. the GA will discover this quickly.

**TWR is a hard constraint**: designs with TWR < 1 are immediately dead. the analytic filter enforces this so the GA never wastes a KSP call on them.

**local optima are real**: a design might be locally optimal (no single mutation improves it) but globally suboptimal (a very different design is much better). this is where crossover and population diversity help — you're not doing hill-climbing from a single point.

---

## the evaluation bottleneck

the GA loop is:

```
generate → validate (cheap) → filter (cheap) → KSP (expensive)
```

KSP is the bottleneck. every KSP call takes ~minutes of real time. this limits how many designs you can evaluate per generation and how many generations you can run.

mitigation strategies:
1. **cheap filters first** — the validator and analytic filters cull bad designs before KSP. a well-tuned filter stack might reject 80-90% of candidates, so KSP only sees the most promising ones.
2. **small population, more generations** — fewer parallel KSP calls but more iterations. useful if KSP can't be easily parallelized.
3. **surrogate model** — train a GNN or other model on accumulated KSP results to predict apoapsis without running KSP. use the surrogate for most evaluations, KSP only for final validation of top candidates. (this is Goal 1+ territory — needs KSP data first.)

for the first implementation: skip the surrogate. use the filter stack to cull hard, run KSP on survivors, collect data, iterate manually. the surrogate becomes viable once you have a few hundred KSP outcomes.

## what we'll build and in what order

### notebook: `notebooks/genetic_algorithm.ipynb`

**step 1 — random rocket generator**
a function `generate_random_rocket(parts_by_name, resource_lookup)` that produces a random structurally valid rocket that passes the analytic filters. this is the initialization step.

**step 2 — fitness function**
`score_rocket(rocket_dict)` — for now, a stub that calls the analytic filters and returns estimated delta-v as a proxy fitness. real KSP fitness comes later.

**step 3 — selection**
`tournament_select(population, scores, k)` — returns a subset of the population selected by tournament.

**step 4 — mutation**
`mutate(rocket_dict, parts_by_name)` — applies one random mutation, validates, returns the result.

**step 5 — crossover**
`crossover(rocket_a, rocket_b)` — stage-level crossover producing one offspring.

**step 6 — the main loop**
```python
population = [generate_random_rocket(...) for _ in range(N)]
for generation in range(max_generations):
    scores = [score_rocket(r) for r in population]
    survivors = tournament_select(population, scores, k=N//2)
    offspring = [mutate(r) for r in survivors] + [crossover(a, b) for a, b in pairs]
    population = survivors + offspring
```

**step 7 — KSP harness integration**
replace `score_rocket` stub with real KSP calls via the harness (this is a later goal — harness not yet built).

## implementation plan — decisions locked in

### fitness function
use `delta_v` (computed by `compute_delta_v`) as the single continuous fitness score. rockets that fail structural validation or analytic filters score **0**. no partial credit for now.

rationale: the cliff at 0 is acceptable because rejection sampling is fast and valid designs will appear in generation 0. partial credit adds complexity and signal dilution — revisit only if the GA fails to bootstrap (i.e. generation 0 has zero valid designs).

```
fitness(rocket):
    if not validate_rocket(rocket):   → 0
    if not filter_rocket(rocket):     → 0
    else:                             → compute_delta_v(rocket)
```

### selection
use **tournament selection**, not fitness-proportionate (roulette wheel). reasons:
- FPS needs a wide continuous fitness distribution to work; early populations will have many 0s and a handful of real scores — FPS degenerates in this regime
- FPS causes premature convergence if one design dominates early
- tournament is simpler, robust, and preserves diversity

### generation 0 (initialization)
random generator with rejection sampling: generate a random rocket, validate + filter, keep if it passes, discard and retry if not. filters are cheap so this is fine.

### crossover
stage-level crossover — take upper stage(s) from parent A, lower stage(s) from parent B. invalid offspring are discarded (free to do because validation is instantaneous).

### build order
1. random rocket generator (`generate_random_rocket`)
2. fitness function stub (`score_rocket` — calls filters, returns delta-v or 0)
3. tournament selection (`tournament_select`)
4. mutation operators (`mutate`)
5. stage crossover (`crossover`)
6. main GA loop
7. KSP harness integration (later — replaces fitness stub with real simulation)
