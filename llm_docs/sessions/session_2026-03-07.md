## what the notebook does

`notebooks/building_the_parser.ipynb` walks through:
1. Reading a raw `.cfg` file
2. Understanding the KSP cfg grammar
3. `classify_line()` — identifies line types (empty, kv, block_name, open, close)
4. `parse_cfg()` — stack-machine parser producing nested dicts
5. `clean_value()` — handles KSP localization strings (`#autoLOC_...`)
6. Field extractors: `extract_engine`, `extract_tank`, `extract_nodes`, `extract_part`
7. Walking the full `Squad/Parts/` directory
8. Inspecting engines, LF+Ox tanks, command parts in tabular form
9. Saving to `data/parts_library.json`

## key decisions (validator)

- checks run in fixed order; `check_part_call` is the gate — if any part type is unknown, later checks don't run
- individual checks return `bool`; final validator will support `verbose=True` for human-readable error list
- `has_minimal_structure` combines root/command/engine checks using `all([...])`
- cycles + connectivity will be done in a single graph traversal starting from the root
- rocket dict format now includes `"stages": {"part_id": stage_number}` — engines and decouplers get stage numbers, pods/tanks do not. stage 0 fires last, higher numbers fire first (KSP convention)
