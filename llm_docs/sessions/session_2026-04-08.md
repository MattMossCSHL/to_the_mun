### craft and install inspection

- inspected the generated craft file and local KSP install
- the generated Terrier and upper tank looked structurally normal in the `.craft`

- however, save-state inspection showed the surviving probe core had:
  - `ElectricCharge amount = 0`

### diagnosis pivot

- this changed the interpretation materially:
  - the upper stage is not failing because the Terrier is absent
  - it is probably not failing because the engine part is malformed
  - it is very plausibly failing because the surviving vessel is an unmanned probe craft that has run out of electric charge before the transfer burn
- since the current GA design space does not include batteries, solar panels, or RTGs, the deeper issue is not just runner behavior
- it is also a design-space mismatch:
  - the GA can currently generate unmanned rockets for a mission class that effectively requires sustained probe power

### project-level decision discussion

- considered two responses:
  - introduce a proper electric-charge analytic model for probe craft
  - or remove unmanned probe roots from the current GA design space
- because the current part vocabulary does **not** include dedicated power-support parts, the robust near-term decision is:
  - do not allow unmanned probe roots in the current `pods` pool
- reason:
  - supporting unmanned Mun-capable rockets without any power-support parts would be dishonest
  - it would keep the GA searching a design region that the current project setup cannot make viable

### robust implementation plan

- decided not to rely on source-path heuristics for crewed vs probe command parts
- instead:
  - extend `src/scraper.py` to extract `CrewCapacity`
  - store `crew_capacity` in `data/parts_library.json`
  - update `src/config.py::load_part_lists(...)` so `pods` includes only command parts with `crew_capacity > 0`
