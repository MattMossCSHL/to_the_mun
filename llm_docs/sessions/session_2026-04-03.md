### conclusion from that run

- the main reasons:
  - `Size2LFB` was only being represented through the generic fallback serializer path

### root cause identified in code

- inspected `src/config.py`
- `load_part_lists(parts_by_name)` currently classifies engines with:
  - `elif part["engine"] is not None: engines.append(name)`
- that is too broad for this project
- any part carrying engine metadata can enter the GA engine pool, including integrated booster / tank+engine parts that the current serializer and rocket model do not actually support well
- `Size2LFB` appears to be one of those bad entrants

### serializer state at end of session

- inspected `src/craft.py`
- unsupported parts still fall through to `make_generic_part_struct(...)`
- `to_craft(...)` then records them as fallback parts and continues
- for critical propulsion parts, that behavior is too permissive because it produces misleading "launchable" craft instead of a hard failure

### recommended next session actions

1. patch `src/config.py` so integrated booster / unsupported propulsion parts do not enter the GA `engines` pool
2. make `src/craft.py` fail explicitly, or at least reject serialization, when a rocket depends on fallback-mode propulsion parts
3. treat any candidate with fallback propulsion parts as invalid for flight testing
